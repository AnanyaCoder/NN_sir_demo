STEP 1 — BUILD REQUIREMENTS
- Single-page app, vanilla HTML/CSS/JS (no build step) unless reference.html already establishes a framework/toolchain — if so, follow it.
- Autoplay by default, timed like a video: each beat holds long enough to read, then transitions to the next. Include play/pause and a scrubber/progress bar along the bottom, plus controls to jump between the four main sections.
- Default pacing pattern: the claim/headline text appears first, then the supporting example/visual rolls in underneath it while the text settles into a caption. Chat-style exchanges (Student/Tutor) should type on turn by turn rather than appear all at once — that's the clearest way to sell "this is a conversation, not a script."
- Include a persistent section index (the 4 sections + closing, listed below) so a viewer can jump straight to e.g. "Section 2 — The Principles."
- Audio: leave hooks for background music/SFX per section (a `data-audio-cue` attribute or similar is enough) but don't block on producing real audio — build muted-by-default with a mute/unmute toggle.
- Responsive down to at least a 1280x720 presentation size; this will be screen-recorded for the final "video," so prioritize looking correct at that resolution.
- Keep transitions simple and consistent (crossfade + slight slide) — don't over-animate; the product screenshots and chat exchanges should read clearly, not fight with the motion.
- The voice input in Section 4/B3 should be a convincing mocked animation (waveform + pre-scripted transcript typing on), not a real microphone/speech-recognition integration — this is a recorded demo, not a live app, so don't request mic permissions or wire up the Web Speech API.

SECTION 2 — THE TUTORBOT PRINCIPLES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Headline: "Every interaction follows evidence-based learning principles."
Present as four sub-beats, each: principle name + one-line definition + a concrete example. Use a consistent card/label treatment across all four so they read as one system, not four different ideas.

Principle 1 — Social Constructivism
Definition: Children learn better by actively participating.
EExample (type on as a chat exchange):
  Student: "Why do we jerk forward when the bus suddenly stops?"
  Tutor: "Before I explain — what was your body trying to do the moment before it stopped?"
Caption line: "The tutor first understands the learner before teaching."

Principle 2 — Dialogic Learning
Definition: Learning happens through conversation.
Example:
  Tutor: "Can you explain this back to me in your own words?"
Caption line: "Rather than checking whether the student memorized the answer, it checks whether they understood it."

Principle 3 — Scaffolding
Definition: Complex problems are broken into manageable steps.
Example — animate as a vertical step sequence, each line appearing only after the last:
  "If we subtract 3, what happens?"
   ↓
  "Great. Now divide by 2."
   ↓
  "Excellent. What's the final answer?"
Caption line: "Instead of giving away the solution, the tutor guides the student one step at a time."
Visual note: if resources/ has a hint-card screenshot (e.g. "HINT – 1 · GENTLE NUDGE" style card), use the real screenshot here as the visual proof point alongside the animated step sequence.

Principle 4 — Activity-Based Learning
Definition: Understanding comes from exploration and interaction.
Visual: An interactive simulation the student can manipulate and observe outcomes from, instead of passively reading explanations. Use a real simulation/interaction screenshot from resources/ if one exists (e.g. the drag-to-explore or Atlas-style interaction); otherwise mock up one clearly interactive example from this list:
  - Move planets to understand seasons
  - Mix colors
  - Change fractions visually
  - Build electric circuits
Pick whichever example best matches an available screenshot; if none match, build the simplest one (color mixing or fraction visualization) as a lightweight interactive mock.

Sub-beat B — Teacher AI
Headline: "Teachers need AI too."

  B1. Differentiated Worksheets
  Teacher prompt (typed into a prompt box, like a real input):
    "Rahul struggles with fractions. Meena is good at fractions but weak in word problems. Generate tomorrow's worksheet."
  Output: two worksheet cards fan out from the single prompt, visibly different (one focused on fraction basics, one on word problems).
  Caption: "The system automatically produces differentiated worksheets tailored to each student."

  B2. Differentiated Lesson Plans
  Teacher prompt:
    "Half my class understands decimals. The other half is still struggling. Plan tomorrow's lesson."
  Output: a lesson plan assembling underneath the prompt, visibly split into two tracks/paths for the two ability groups.
  Caption: "AI generates a lesson plan that supports mixed-ability classrooms."

  B3. Administrative Assistant (voice input)
  Open on a mic button/voice UI, not a text box — this beat should read as "the teacher just talks, the AI does the rest."
  Sequence:
    1. Teacher taps a mic icon; a waveform animates while "speaking."
    2. A live transcript types on underneath the waveform, line by line, as if being transcribed in real time:
       "Rahul had difficulty with reading today."
       "Priya improved in multiplication."
       "Arjun was absent."
    3. Once the transcript settles, it visibly hands off to the AI (e.g. waveform/mic fades, a small "generating" state plays briefly).
    4. Output: three separate personalized parent write-up cards appear, one per student — not a single generic summary. Each should read like a warm, specific note a teacher would actually send, not a template with names swapped in:
       - Rahul's card: acknowledges the reading difficulty gently, frames it as something being worked on, no alarm tone.
       - Priya's card: highlights the multiplication improvement specifically and positively.
       - Arjun's card: notes the absence plainly and neutrally, inviting a quick check-in if needed.
    5. Render at least one of the three (pick whichever reads most naturally) in genuine code-mixed language (e.g. Hindi-English) rather than plain English, to demonstrate multilingual capability on real generated output — this is the one place in the film that should explicitly show code-mixed output, so don't render everything in plain English.
  Caption: "Teachers speak naturally. TutorBot writes what each parent actually needs to hear."
